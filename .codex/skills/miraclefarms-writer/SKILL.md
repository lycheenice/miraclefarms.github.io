# MiracleFarms Writer

给定主题 + 参考链接 -> 自动调研 -> 产出可直接发布到 miraclefarms.github.io 的 `_posts/` 文章。

---

## 第一步：判断文章类型

在开始调研之前，先确认写哪种文章：

**Essay（技术深度分析）**
- 主题是深度技术分析、论文解读、架构分析、源码解读、benchmark 综述
- 通常 800–3000 字，结构严密，有核心论点
- 作者：Ethan

**Brief（日报 / 简报）**
- 主题覆盖多个项目过去 24–48 小时的更新进展
- 或用户明确说"日报/简报/brief"
- 作者：荔枝不耐思

**如果用户没有明确说明类型**，根据参考资料判断：
- 参考资料以 PR 列表为主 -> brief
- 参考资料是一篇论文 / 单一技术主题 -> essay
- 仍不确定 -> 直接问用户

---

## 第二步：调研

**对每个提供的参考 URL：**

- **GitHub PR**：读取 PR 标题、description、comment、changed files（重点看 diff 摘要）。提取：做了什么、解决了什么问题、影响了哪个模块。
- **arXiv 论文**：读取摘要、Introduction、核心贡献段落、实验结论。重点提取：核心创新是什么、和现有工作的差异、关键数字结论。
- **博客 / 官方文档**：读取全文，提炼核心观点和数据。
- **本地 _posts/ 文章**：直接 read 文件，作为背景参考。

**证据可信度优先级**（影响你可以做多强的判断）：
1. 论文原文（arXiv / conference）
2. 官方 PR / commit（已合并）
3. 官方 Release / 官方博客
4. 官方文档（README / API docs）
5. 新闻报道 / 二手转述

**确定核心判断**：在写作之前，先问自己——"这批材料最想说明什么？有没有一个能贯穿全文的核心观点？"这个判断是文章的骨架，不是材料的汇总。

如果参考资料不足以支撑某个具体判断，要在文中标注"尚待验证"或"基于推断"。

---

## 第三步：写作原则（Brief / Essay 通用）

### 开篇立旗，而不是铺背景

文章第一段必须是核心判断或核心问题——不是"X 项目最近发布了 Y"，而是"这波更新说明了什么"。读者读完第一段就知道这篇文章的中心是什么。

### 叙述，而不是列点

关键概念用段落展开，用"首先 / 其次 / 因此 / 但是"等过渡语连接。避免把三个 PR 变成三个 bullet。如果有并列事实，把它们组织成有逻辑递进的叙述。

### 有观点，有边界

- 允许第一人称判断："作者认为……""这里有一个值得关注的问题……""但这个结论有一个前提……"
- 明确判断的适用边界：在什么条件下成立、什么情况下还不确定
- 不要假装客观中立地列事实；要有主线判断

### 避免的句式

- ❌ "不是 X，而是 Y"——可以偶用，但别在同一篇文章里出现三次以上
- ❌ "值得注意的是……"——句子开头直接说结论
- ❌ "综上所述，我们可以得出……"——结论段直接写结论，不要过渡公式
- ❌ 每个 PR / 每个模块单独一段流水账，失去全文贯穿的主线

### 标题编号

- H2 用中文数字顿号：`## 一、` `## 二、` `## 三、`
- H3 可用阿拉伯数字：`### 1.1` `### 1.2`
- 正文不用 H1（`#`），front matter 的 `title` 会自动渲染成页面 H1

---

## 第四步：格式规则

**根据文章类型，读取对应的格式规范文件：**

- **Brief** -> 读 `references/brief-format.md`（frontmatter 模板、引用格式、完整结构模板）
- **Essay** -> 读 `references/essay-format.md`（frontmatter 模板、引用格式、完整结构模板）

这两个文件包含所有格式细节，包括完整模板。**写作前必须先读对应文件。**

---

## 第五步：输出文件

- 保存路径：`/Users/lychee/mycode/miraclefarms.github.io/_posts/YYYY-MM-DD-slug.md`
- 文件名：日期取自 front matter 的 date 字段 + 英文 slug（小写、连字符分隔）
  - 例：`2026-04-15-vllm-paged-attention-design.md`
  - 例：`2026-04-15-ai-infra-daily-brief-speculative-decoding.md`
- **不要主动 commit 或 push**，除非用户明确要求

---

## 图片处理

- 如果参考材料有可引用的图（论文 figure、官方架构图、GitHub 截图），可以提示用户手动下载或建议下载路径
- 图片存放在：`/assets/{post-slug}/`
- 引用格式：`![描述](/assets/slug/fig-1-xxx.png)` + `*图 N：说明文字。*`
- Essay 文章建议配图；Brief 通常不需要

---

## 快速检查清单

写完后自检：

- [ ] front matter 所有必填字段都有？
- [ ] `kind` 和 `category` 配对正确？
- [ ] 日期用 `+0800`？
- [ ] Brief：引用是 `[[N]](url)` 格式？有 `## 参考来源` 章节？
- [ ] Essay：引用是 `<a href="url">[N]</a>` 格式？有 `## 参考资料` 章节？
- [ ] 标题用了全角竖线 `｜`（Brief）或无竖线（Essay）？
- [ ] H2 用了中文数字编号？
- [ ] 开篇是判断，不是背景介绍？
- [ ] 没有满篇 bullet list，是连续叙述？
