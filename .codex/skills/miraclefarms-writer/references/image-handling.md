# 图片处理参考

参考资料里出现图片时，目标不是“尽量多放图”，而是“尽量拿到最能支撑文章论点的原图”。

## 1. 选图原则

- 只选和正文主线直接相关的图：架构图、方法流程图、关键实验图、实现前后对比图。
- 优先官方来源：论文原图、repo 原图、官方博客正文图。
- 跳过装饰图：logo、封面 hero、作者头像、营销 banner、与技术判断无关的 UI 截图。
- 同一论点通常只需要一张图；如果两张图表达的是同一个意思，选更清晰、信息密度更高的那张。

## 2. 原图优先顺序

按下面顺序尝试，前一步成功就不要退化到后一步：

1. 页面中直接引用的原始图片 URL
2. `srcset` / `picture` 中分辨率最高的版本
3. GitHub raw 文件或 `user-attachments` 原图
4. arXiv HTML 的 figure 资源
5. 论文 PDF 裁图或页面截图

最后一项只是兜底，不是默认方案。

## 3. GitHub / Repo 图片

- 对 README、文档页、PR 描述中的图片，优先找 markdown 原始链接或页面里的直接资源 URL。
- GitHub 文件链接如果是 `blob/...`，优先换成 `raw.githubusercontent.com/...` 的原图地址。
- 对 PR / issue 上传图，优先保留 `github.com/user-attachments/...` 或其他 GitHub 托管的原始资源链接。
- 如果同一图片既有缩略图又有原图，选原图。

## 4. Blog / 官方文档图片

- 优先正文里的 `<figure>`、`<img>`、`<picture>`、`srcset`，不要默认拿 `og:image`。
- 如果站点给了多个尺寸，优先选正文实际引用的最大版本。
- 只有当 `og:image` 本身就是正文核心图时，才把它当候选；不要把文章封面误当成技术插图。
- 如果图片 URL 带明显缩略参数，只有在规则足够明显时才尝试去掉缩放参数；不要猜测复杂 CDN 规则。

## 5. arXiv / Paper 图片

- 论文优先使用 arXiv HTML 版本中的 figure 资源，不要默认从 PDF 截图。
- 先找 `https://arxiv.org/html/<paper-id>` 或论文页面指向的 HTML 版本，再定位 `<figure>` 和其图片资源。
- 如果 HTML figure 能拿到原始 PNG / JPG / SVG，就直接下载该资源，并保留 figure caption 作为图注素材。
- 如果 arXiv HTML 不可用、图资源 403、或 HTML 里只有低质量预览，再考虑其他官方 HTML 镜像。
- 只有在原图不可得但这张图又对正文非常关键时，才回退到 PDF 裁图；此时优先裁出完整 figure，而不是整页截图。

## 6. 失败回退策略

抓图失败时按顺序回退：

1. 换同页里更直接的原始资源 URL
2. 换同一来源的另一张等价图
3. 换同主题的官方替代图
4. 对论文使用 PDF 裁图
5. 放弃插图，改用正文解释

不要因为单张图失败就卡住整篇文章。

## 7. 落盘与插入

- 路径：`/assets/{post-slug}/`
- 文件名：`fig-{序号}-{简短英文语义}.{ext}`
- 优先保留原始格式；如果来源是 SVG，就不要无谓转成 PNG。
- 图片插入位置应紧跟对应分析段落，避免图文分离。
- 图注要写“这张图为什么重要”，必要时在句末补充来源，例如“来源：arXiv HTML figure”。

示例：

```markdown
![PagedAttention 页表结构示意图](/assets/vllm-paged-attention-design/fig-1-page-table.svg)

*图 1：PagedAttention 用页表把逻辑 token 块映射到非连续显存页，因此把 KV cache 扩容问题从连续分配改成了间接寻址。来源：论文原图。*
```

**注意**：图片行和图注行之间必须有一个空行，否则 Jekyll/CommonMark 会把它们渲染在同一段落里，图注紧贴图片下方而不换行。

## 8. 写作边界

- 不要为了版面好看而插图。
- 不要自己重绘原图并伪装成来源图片。
- 如果图的关键结论已经能用一句话讲清，且抓取成本很高，可以不插。
- 如果图片版权或可访问性明显不稳定，宁可少一张图，也不要把不可靠的下载流程写进正文依赖里。
