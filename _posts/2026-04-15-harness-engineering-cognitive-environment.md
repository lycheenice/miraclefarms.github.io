为什么在模型能力持续提升的今天，生产级 Agent 系统的可靠性差异与模型本身关系越来越小？当 OpenAI Codex、Anthropic Claude Code 等系统在相似规模的模型上 converge 出高度相似的架构模式时，核心差异已经不在模型内部，而在于包裹模型的运行时结构。这指向一个被忽视但正在改变游戏规则的设计哲学：驾驭工程（Harness Engineering）。可靠的 Agent 系统不是模型加上外围工具，而是模型嵌入到精心设计的认知环境中——这个环境通过六个维度的结构性约束，将未限定的推理任务转化为模型可以可靠执行的结构化操作<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。

## 一、从认知外化到驾驭工程

人类认知史可以读作外化史：语言把私密思想变成共享符号，写作把脆弱记忆变成持久记录，印刷术把知识复制变成社会机器，数字计算把算术劳动变成可编程设备。每一次转变的关键都在于"代表转换"（representational transformation）：工具不是简单地放大原有能力，而是重构问题本身，让内部能力在新的问题结构下更可靠地发挥作用<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。购物清单不是扩容生物记忆，而是把回忆问题变成识别问题；地图不是让导航"更强"，而是把隐式空间关系变成可见结构。

LLM Agent 领域正在经历同样的外化逻辑。早期系统期望模型从权重内部恢复所需能力；后来的系统把一部分能力外化到上下文；最新的系统则把能力进一步外化到持久化的外部结构中：Memory 外化了跨时间状态，Skills 外化了过程性专业知识，Protocols 外化了交互结构，而 Harness 则作为统一层，协调这些模块在受控的运行时中可靠地执行<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。

这个转变的核心观点是：可靠智能体能力不完全来自更大的模型，而来自系统性地重构任务需求，使内部能力和外部基础设施联合覆盖完整的竞争力谱系。这解释了为什么生产系统收敛在相似的架构模式上——这些模式不是偶然的实现选择，而是外化智能的结构性要求。

## 二、什么是 Harness：从脚手架到认知环境

Harness 是那个"包裹模型的运行时"。它不是一系列附加工具的集合，而是设计的认知环境，在这个环境中，外化模块（Memory、Skills、Protocols）才能协同生效<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。一个实用的 Agent 等于"模型在 Harness 中运行"，而不是"模型带着外围能力"。

![Harness 作为认知环境的六维度结构](/assets/harness-engineering-cognitive-environment/fig-7-harness-cognitive-environment.png)
*图 1：Harness 作为认知环境的六维度结构。外围的三项外部化模块（Memory、Skills、Protocols）提供内容，内部的三个操作面（Permission、Control、Observability）提供治理，通过连续的循环流动协调执行。来源：论文原图。*

Harness 的六个维度可以分成两层：内容层承载能力，治理层约束执行。Memory 提供状态持久化、失败记录、跨会话上下文；Skills 提供可复用例程、分阶段加载、失败驱动的修订；Protocols 提供确定性接口、结构化调用、模式契约。Permission 通过沙箱、文件隔离、网络限制定义边界；Control 通过递归边界、成本上限、超时约束执行；Observability 通过结构化日志、执行追踪、聚合度量提供可见性。

关键在于，Har ness 不会把这些维度作为简单的开关提供，而是作为精心设计的环境结构：默认值、钩子、文件边界、技能调用模式、检查门控，所有这些结构规律都在缩小 plausible action 的空间。这把未限定的任务变成了"引导行动的结构化环境"——这正是 Norman 所说的认知Artifact的核心功能<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。

## 三、驾驭工程的六个维度

Agent Loop 和控制流构成了 Harness 的时间骨架。Perceive → Retrieve → Plan → Act → Observe 的循环将无限可能折叠为一个清晰的操作序列。终止控制——最大步数、递归深度边界、资源上限、超时约束——不是对模型不信任的折衷，而是定义推理展开的操作边界。一个调优良好的循环通过限制可能的执行路径空间让 Agent 更可靠，而不是让模型变得更聪明<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。

沙箱和执行隔离作为认知边界运作。专用云沙箱、文件系统快照、网络限制、资源配额不仅制造安全围栏，还简化了 Agent 的操作环境：移除不相关状态，让工作空间可被检查。毕业式权限模式（autonomous → approval_required）进一步确认了边界的认知性质——它改变了模型必须推理的内容，而不是仅仅控制它能触及什么<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。

人类监督和检查门控揭示了自主性不是二元属性，而是 Harness 的可配置参数。三种常见模式——执行前批准、执行后审查、升级触发——让自主性可以按任务、工具、组织策略调整。钩子系统允许操作者在生命周期事件（工具调用、文件写入、子 Agent 生成）上附加任意逻辑，把治理变成了可编程的过程<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。

可观测性和结构化反馈服务于两个目的。对外，它支持调试、合规审计、事后分析；对内，它闭合反馈循环——失败的工具调用写入 Memory，重复失败触发 Skill 修订。这不是辅助便利，而是 Harness 从自身操作中学习的机制<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。

配置、权限和策略编码把原本嵌入在 Prompt 或在事后过滤中强制执行的约束，变成了运行时执行的声明式规则。用户级偏好、项目级工具集、组织级合规要求形成了分层配置，让治理可以在不同粒度上表达<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。

上下文预算管理把最稀缺的共享资源（上下文窗口）在 Memory 检索、Skill 加载、Protocol 模式、工具描述、推理追踪之间动态分配。总结化、基于优先级的驱逐、分阶段加载等策略的正确性取决于 Agent 当前目标、正在使用的模块和执行约束，因此这不仅是一个压缩问题，而是一个关于资源分配的动态调度问题<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。

## 四、实践中的收敛模式

当代生产 Agent 系统（OpenAI Codex、Anthropic Claude Code）在 Harness 结构上收敛出了惊人的相似性：显式循环+终止控制、作为认知边界的沙箱、带钩子的可配置检查门控、闭合内部反馈循环的结构化追踪、分层配置、主动的上下文预算管理<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。这种收敛的意义在于：主要设计挑战不是从模型中提取更好的完成，而是安排操作条件，让完成变成有效的干预。

这暗示了一个深层判断：可靠的智能体能力是一个生态成就（ecological achievement），而不是单体属性。它产生于 Agent 被嵌入到一个认知环境中，这个环境的组织结构让有益行为更容易，让有害行为更困难。一致性不是巧合，而是对结构要求的各种系统的平行发现。

## 五、三个外部化模块的协同

Harness 作为协调层，让 Memory、Skills、Protocols 三者形成双向强耦合。

![Memory、Skills、Protocols 之间的耦合关系](/assets/harness-engineering-cognitive-environment/fig-8-couplings-memory-skills-protocols.png)
*图 2：三个外部化模块通过六条双向流动强化彼此。Memory→Skill 是经验蒸馏（轨迹变成可复用过程）；Skill→Memory 是执行记录（生成验证可靠性的追踪）；Skill→Protocol 是能力调用；Protocol→Skill 是能力生成；Memory→Protocol 是策略选择；Protocol→Memory 是结果同化。这些流动形成了 Harness 内的系统级动态。来源：论文原图。*

经验上的证据支持了这个框架。OpenAI Codex 的讨论中多次提到 Harness 如何通过 Memory 集成失败模式，通过 Skill 组织可复用例程，通过 Protocol 标准化与代码库的交互<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。Anthropic Claude Code 则展示了分层配置如何在不同粒度编码权限，可观测性追踪如何闭合反馈循环，上下文预算管理如何在长会话中维持性能<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。

这些模式的重复出现强化了一个核心判断：Agent 系统的能力是分布式的——分布在模型参数、外部 Memory 存储、可执行 Skills、Protocol 定义、工具表面、监控系统和运行时约束之间。Harness 是协调这个分布式系统的介质<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。

## 六、为什么驾驭工程重要

驾驭工程的重要性在于它提供了一个重新分配智能的理论框架。可靠智能体不来自更大的模型，而来自任务需求的系统性重构——让内部能力和外部基础设施联合覆盖完整的竞争力谱系<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。

这改变了边界问题的表述方式。关键设计问题不是智能属于模型还是基础设施，而是给定更新频率、重用模式、治理需求和执行成本，特定负担应该在哪里<a href="https://arxiv.org/html/2604.08224v1">[1]</a>。频繁变化的策略适合放在 Prompt 中；相对稳定的技能可以外部化为可复用例程；需要跨会话保持的知识应该存入 Memory；共享的交互接口应该标准化为 Protocol；执行边界应该在沙箱中定义；治理约束应该在配置中编码。

这个判断的一个推论是：Agent 系统的可靠性来自架构转换，而非单纯的能力提升。Har ness 通过把未限定的任务变成结构化环境，把回忆变成识别，把即兴生成变成组合，把临时协调变成结构合约，把模型需要处理的问题结构转换成它可以更可靠地解决的形式。

另一个推论是：只有当外部化模块被嵌入到具有可观测性和反馈循环的 Harness 中时，它们才能实现自我修正和适应。这解释了为什么生产系统都收敛于相似的观察和追踪模式——没有它们，外部化模块无法从执行中学习，无法从经验中演化。

## 七、结论

驾驭工程的兴起标志着 LLM Agent 设计从模型中心到环境中心的转移。可靠的智能体能力不完全来自更大的模型，而来自认知环境的系统性设计。Harness 不是给模型添加外部能力，而是设计一个让模型可以更可靠地发挥作用的环境结构。

这个判断在实践上的意义深远。评估 Agent 系统时，不应只看模型参数规模，而应看 Harness 的六个维度如何组织；优化 Agent 系统时，首要工作不是升级模型，而是精化环境结构；架构决策时，首要考虑不是某个能力应该放在哪里，而是它的更新频率、重用模式、治理需求和执行成本如何决定它的正确位置。

未来的方向已经隐约可见：自进化的 Harness（self-evolving harnesses）、共享的 Agent 基础设施、模型与外部基础设施的长期协同演化，都是当前趋势的自然延续。当 Harness 本身开始演化学习，当多个系统共享相似的结构模式，当模型和基础设施相互塑造，智能体能力的边界将继续扩展——不是通过单体模型的无限制增长，而是通过分布式系统的持续重组。
